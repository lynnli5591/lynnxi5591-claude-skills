# 需求拆解与产品架构

## 输入来源

需求可能来自：
- **业务截图**：现有系统界面、Excel 表格、流程图
- **文字描述**：业务场景说明、痛点描述
- **访谈记录**：与业务方的沟通内容

## 第一步：识别业务本质

拿到输入后，先回答三个问题：

1. **这个系统管理的是什么对象？**（人、钱、货、申请单、规则……）
2. **核心业务流程是什么？**（谁发起 → 谁审核 → 谁执行 → 产生什么结果）
3. **有哪些约束条件？**（金额上限、时间截止、资格准入、预算管控……）

## 第二步：分层架构

将业务拆解为 A-F 六层（以薪资管理为例，可类比到其他业务）：

```
A 上游数据层   — 数据从哪来，质量如何保证
B 管控规则层   — 有哪些限制和阈值（预算、额度、资格）
C 业务规则层   — 核心业务逻辑（档次、等级、分类）
D 申请/发起层  — 谁来发起，发起的条件是什么
E 校验/审核层  — 系统自动校验 + 人工复核
F 执行/输出层  — 最终结果（发放、记录、报表）
```

**输出格式**：用表格呈现，每层列出：层名、职责描述、关键功能点、数据来源。

## 第三步：识别关键约束与流转规则

重点梳理：

- **阻断关系**：A 不完成，B 无法进行（如：校验异常未处理，申请不放行）
- **触发关系**：满足条件 X 时，自动触发 Y（如：超额 5% 触发审批）
- **级联关系**：修改 A 影响 B（如：档次规则变更影响人员匹配）

用流程图文字描述清楚：
```
数据采集 → 校验（管理员处理异常）→ 放行 → HR 复核 → 核算发放
```

## 第四步：功能清单

每个分层内细化功能点，格式：

| 功能名称 | 所属层 | 归属角色 | 优先级 | 说明 |
|---|---|---|---|---|
| 薪资档次划分 | C层 | 管理员 | P0 | 规则配置入口 |
| 人员档次匹配 | C层 | 管理员 | P0 | 支持自动+手动 |
| 申请复核 | E层 | HR专员 | P0 | 仅看干净数据 |

## 常见陷阱

- **不要把校验异常处理和申请复核混在一个角色**：数据问题归数据专员/管理员，业务审批归 HR
- **不要把配置功能给普通用户**：预算设定、规则配置是管理员权限
- **注意上下游依赖**：某些功能必须等上游完成才能操作

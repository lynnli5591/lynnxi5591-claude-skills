# 阶段 2：架构设计与角色权限

## 角色设计原则

### 原则一：从「谁处理异常」反推权限边界

权限设计最容易出错的地方是「异常情况」。先问：
- 数据出错了谁来修？（通常是管理员/数据专员，不是业务用户）
- 审批卡住了谁来推？（取决于审批层级设计）
- 规则配错了谁来改？（有配置权限的管理员）

把这些「异常处理者」的权限设计清楚，正常流程的权限自然就清晰了。

### 原则二：业务用户只看「干净数据」

HR、业务专员等非技术角色，不应该看到：
- 系统校验的原始异常（这是数据层的问题）
- 其他角色的操作队列
- 配置类功能（规则、预算设定）

他们应该看到的是：**已经过处理、可以直接操作的业务数据**。

### 原则三：「只读提示」vs「可操作入口」

同一个信息，不同角色的展示方式不同：

| 信息 | 管理员看到 | HR 看到 |
|---|---|---|
| 校验异常 2 条 | 可点击处理，有操作按钮 | 「管理员处理中」只读提示 |
| 预算设定 | 可编辑金额、阈值 | 只看余额和使用率 |
| 规则配置 | 可增删改 | 不可见 |

## 侧边栏菜单设计

### 分区原则

```
工作台         → 当前角色最常用的操作入口
[业务模块1]    → 按业务流程顺序排列
[业务模块2]
数据/配置      → 基础档案、规则配置等低频操作
```

### 菜单放在哪个角色下的判断标准

问三个问题：
1. **谁配置它？** → 配置者拥有这个菜单
2. **谁使用它？** → 使用者能看到（可能只读）
3. **它属于「规则」还是「操作」？** → 规则归管理员，日常操作归业务用户

### 导航层级设计

- 顶部导航：区分不同角色/模块（工作台 / 规则配置）
- 侧边栏：当前模块下的页面列表
- Tab：同一页面下的视图切换

**原则：能用 Tab 解决的，不要开新菜单；能共用侧边栏的，不要加顶部导航。**

## 模块间的状态流转设计

这是最容易被忽视但最重要的设计。

### 流转图设计方法

```
[上游模块产出] → [本模块输入] → [本模块产出] → [下游模块消费]
     ↓                                                  
  如果异常 → [异常处理者] → [处理结果] → [重新进入流程]
```

### 关键设计问题

1. **放行条件**：什么条件下，数据才能从 A 模块流向 B 模块？
2. **拦截逻辑**：什么情况下流程被中断？谁来处理？处理完后如何恢复？
3. **状态可见性**：每个角色能看到流程的哪个阶段？

### 常见状态流转模式

**串行审批模式**：
```
申请 → 校验（系统）→ 异常处理（管理员）→ 业务复核（HR）→ 核算发放
```

**并行处理模式**：
```
申请 → 自动校验 + 预算检查（并行）→ 都通过后进复核
```

**层级审批模式**：
```
超额 ≤5% → HR负责人审批
超额 5~20% → 总监审批  
超额 >20% → C级审批
```

## 数据关联设计

### 实体关系梳理

在设计页面前，先列出核心实体及其关联：

```
档次 ─── 关联多个 ───▶ 薪资项目
  │
  └── 匹配多个 ───▶ 人员
        │
        └── 产生 ───▶ 申请
              │
              └── 触发 ───▶ 校验 → 复核 → 核算
```

### 关联的实现方式

- **编码关联**：用唯一编码（SAL-001）做外键
- **名称冗余**：列表展示时冗余名称，避免频繁跨表查询
- **状态同步**：关联数据变更时，相关状态要联动更新

## 典型角色模板

### 管理员/数据专员
**核心职责**：配置规则、处理数据异常、审批流管理  
**常用功能**：校验规则库、异常处理、预算设定、档次配置  
**不做的事**：发起业务申请、直接操作员工数据  

### HR 专员/业务用户
**核心职责**：业务审核、申请复核、数据查看  
**常用功能**：工作台、申请复核、预算追踪、核算明细  
**不做的事**：修改规则配置、处理数据层异常  

### 员工/申请人
**核心职责**：发起申请、查看状态  
**常用功能**：申请入口、进度查询  
**不做的事**：看到其他人的数据、修改审核结果

# Product Prototype Skill

将业务需求转化为**可交互 HTML 原型**的完整工作流，覆盖终端确认、产品思维、架构设计、角色分工、原型构建、迭代优化全链路。

---

## 第零步：确认终端类型（必须，不可跳过）

**在做任何架构设计之前，必须先明确产品的终端类型。**

如果用户没有明确说明，必须让用户从以下选项中选择一个，不能自行假设：

| 终端类型 | 适用场景 | 原型风格 |
|---|---|---|
| **移动端 App** | iOS / Android 手机应用 | 底部 Tab、页面层级、手势交互 |
| **PC 管理后台** | 桌面端管理系统 | 左侧导航、表格/表单、弹窗 |
| **网页 / H5** | 浏览器访问的网页产品 | 顶部导航、响应式、转化路径 |
| **小程序** | 微信 / 支付宝小程序 | TabBar、轻量化页面、微信生态 |

**判断规则：**
- 用户说"App"、"应用"、"手机端" → 移动端 App
- 用户说"后台"、"管理系统"、"dashboard" → PC 管理后台
- 用户说"网站"、"官网"、"H5" → 网页 / H5
- 用户说"小程序"、"微信端" → 小程序
- **描述模糊时（如只说"系统"、"平台"）→ 必须询问用户确认，给出 4 个选项让用户选择**

⚠️ **终端类型直接决定架构设计视角和原型风格，错误的终端类型会导致整个原型方向错误。**


## 快速开始

### 判断输入类型

```
有截图/文档 → 读 references/01-requirements.md，进行需求拆解
纯文字描述 → 直接架构设计，输出分层结构
已有原型   → 直接进入 references/04-iteration.md
```

### 必须在写代码前完成的事

**第一步：确认终端类型**（见第零步）

**第二步：输出产品架构给用户确认**，包含：
- 功能模块列表（分层结构，结合终端特性）
- 角色列表及各角色核心任务
- 关键业务流程（数据流向、审批链路）

**不要跳过这两步直接出原型。**

## 终端类型对架构设计的影响

### 移动端 App
- 架构重点：底部导航 Tab 划分、页面层级（主页→列表→详情）、返回逻辑
- 原型规范：宽度 390px，顶部标题栏，底部 TabBar，iOS 圆角卡片风格，主色 #1890ff
- 交互重点：点击列表项进详情，详情页有返回按钮，按钮有点击反馈

### PC 管理后台
- 架构重点：左侧多级菜单、功能模块划分、权限层级、批量操作
- 原型规范：左侧导航栏（深色 #001529），右侧内容区，白色卡片，主色 #1890ff
- 交互重点：表格行点击进详情，弹窗表单，状态筛选，分页

### 网页 / H5
- 架构重点：用户转化路径、核心页面流程、SEO 结构、响应式断点
- 原型规范：顶部导航，内容居中（最大宽 1200px），面包屑导航
- 交互重点：页面间跳转，表单提交，状态反馈

### 小程序
- 架构重点：TabBar 页面划分（最多 5 个）、轻量化功能、微信生态集成
- 原型规范：宽度 375px，底部 TabBar，主色 #07C160（微信绿），符合小程序规范
- 交互重点：页面栈管理，轻量弹窗，微信授权流程

## 核心原则

1. **终端优先**：终端类型确认前不做任何架构设计
2. **先思考后动手**：架构确认前不写代码
3. **精准修改**：迭代时只改必要的地方
4. **角色视角清晰**：每个功能明确归属哪个角色
5. **逻辑自洽**：模块间关系要前后一致
6. **语法验证**：每次改动后验证 JS 语法

---

## 输出规范

- **格式**：单文件 HTML（含内联 CSS + JS），无外部依赖
- **页面切换**：用 JS 函数控制 `display` 切换，禁止用 `<a href>` 跳转
- **设计语言**：根据终端类型选择对应风格（见上方终端规范）
- **文件命名**：`{系统名}_prototype_v{版本号}.html`
- **版本标注**：顶部显示版本号
- **二级页面**：必须支持进入详情页并能返回，不能只有一层页面
